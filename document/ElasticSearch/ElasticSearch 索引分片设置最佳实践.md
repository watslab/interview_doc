# ElasticSearch 索引分片设置最佳实践

## 概述

分片（Shard）是 ElasticSearch 实现分布式特性的核心机制。合理的分片设置对集群性能、稳定性和可扩展性至关重要。本文档详细介绍分片设置的最佳实践及其背后的原因。

---

## 一、分片基础概念

### 1.1 什么是分片？

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        分片结构示意                                       │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   Index（索引）                                                          │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │                          数据总量: 150GB                         │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                              │                                          │
│                              ▼                                          │
│                    分片数量: 5 个主分片                                   │
│                              │                                          │
│        ┌──────────┬──────────┼──────────┬──────────┐                   │
│        ▼          ▼          ▼          ▼          ▼                   │
│   ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐         │
│   │ Shard 0 │ │ Shard 1 │ │ Shard 2 │ │ Shard 3 │ │ Shard 4 │         │
│   │  30GB   │ │  30GB   │ │  30GB   │ │  30GB   │ │  30GB   │         │
│   └─────────┘ └─────────┘ └─────────┘ └─────────┘ └─────────┘         │
│                                                                         │
│   【说明】                                                               │
│   • 每个分片是一个独立的 Lucene 索引                                     │
│   • 分片可以分布在不同的节点上                                           │
│   • 分片数量在索引创建时指定，之后不可修改                                │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 1.2 分片类型

| 类型 | 说明 | 作用 |
|------|------|------|
| **主分片（Primary Shard）** | 原始数据分片 | 数据存储、读写操作 |
| **副本分片（Replica Shard）** | 主分片的复制 | 高可用、读负载均衡 |

### 1.3 分片的重要特性

| 特性 | 说明 |
|------|------|
| **不可变性** | 主分片数量在索引创建后不可修改 |
| **分布式** | 分片可以分布在集群的任意节点上 |
| **独立性** | 每个分片是一个独立的 Lucene 索引 |
| **可复制** | 每个主分片可以有多个副本 |

## 二、分片设置最佳实践

### 2.1 核心原则（黄金法则）

| 原则 | 建议值 | 原因 |
|------|--------|------|
| **单分片大小** | 10GB - 50GB | 平衡性能与恢复速度 |
| **单分片文档数** | 不超过 2 亿文档 | 避免单个分片过大 |
| **每 GB 堆内存分片数** | 不超过 20 个 | 避免内存压力过大 |
| **单节点分片数** | 不超过 1000 个 | 避免资源耗尽 |

### 2.2 分片大小建议详解

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     分片大小与性能关系                                    │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   分片大小        性能表现                                               │
│                                                                         │
│   < 1GB          ⚠️ 过小                                                 │
│   ├── 问题：管理开销大、资源浪费                                         │
│   └── 场景：仅适用于小规模企业搜索                                       │
│                                                                         │
│   1GB - 10GB     ✅ 较小（企业搜索适用）                                  │
│   ├── 优点：恢复快、并行度高                                             │
│   └── 缺点：管理开销较大                                                 │
│                                                                         │
│   10GB - 50GB    ⭐ 最佳范围（推荐）                                      │
│   ├── 优点：性能与恢复速度平衡                                           │
│   ├── 恢复：可在合理时间内完成                                           │
│   └── 适用：大多数业务场景                                               │
│                                                                         │
│   50GB - 100GB   ⚠️ 较大                                                 │
│   ├── 问题：恢复时间长、GC 压力大                                        │
│   └── 场景：网络性能好的环境可考虑                                       │
│                                                                         │
│   > 100GB        ❌ 过大（不推荐）                                        │
│   ├── 问题：恢复极慢、故障恢复困难                                       │
│   └── 风险：单点故障影响大                                               │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.3 分片数量计算公式

```
分片数量 = 预估数据总量 / 单分片目标大小

示例：
• 预估数据总量：500GB
• 单分片目标大小：30GB
• 分片数量 = 500 / 30 ≈ 17 个主分片

建议向上取整，预留扩展空间：20 个主分片
```

## 三、分片过多的问题（Oversharding）

### 3.1 什么是分片过多？

当集群中分片数量超过集群资源的承载能力时，称为分片过多（Oversharding）。

### 3.2 分片过多的危害

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     分片过多的危害                                        │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   1. 搜索性能下降                                                        │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │ • 每个搜索请求需要访问多个分片                                    │   │
│   │ • 协调节点需要合并多个分片的结果                                  │   │
│   │ • 分片越多，合并开销越大                                          │   │
│   │ • 查询延迟增加                                                    │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│   2. 资源开销增加                                                        │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │ • 每个分片需要占用内存（Segment、FST 等）                         │   │
│   │ • 每个分片需要 CPU 资源进行管理                                   │   │
│   │ • 每个分片需要文件描述符                                          │   │
│   │ • 堆内存压力增大，GC 频繁                                         │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│   3. 集群稳定性风险                                                       │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │ • Master 节点管理负担加重                                         │   │
│   │ • 集群状态更新变慢                                                │   │
│   │ • 极端情况下可能导致集群崩溃                                      │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│   4. 恢复时间延长                                                        │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │ • 节点重启后需要恢复大量分片                                      │   │
│   │ • 分片分配和恢复耗时增加                                          │   │
│   │ • 影响业务可用性                                                  │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 3.3 分片过多的具体影响

| 影响项 | 说明 |
|--------|------|
| **内存消耗** | 每个分片约占用 10-50MB 堆内存 |
| **文件描述符** | 每个分片需要数百个文件描述符 |
| **CPU 开销** | 分片合并、查询都需要 CPU |
| **网络开销** | 分片间通信增加网络负载 |
| **GC 压力** | 分片多导致对象多，GC 频繁 |

### 3.4 分片过多的判断标准

```json
// 查看集群分片数量
GET /_cat/shards?v&h=index,shard,prirep,state&s=index

// 查看每个节点的分片数量
GET /_cat/allocation?v&h=shards,disk.indices,disk.used,disk.avail,node

// 判断标准
// 1. 单节点分片数 > 1000：需要优化
// 2. 每 GB 堆内存分片数 > 20：需要优化
// 3. 集群状态更新延迟 > 1s：需要优化
```

## 四、分片过少的问题

### 4.1 分片过少的危害

| 问题 | 说明 |
|------|------|
| **资源浪费** | 无法充分利用集群的计算和存储资源 |
| **性能瓶颈** | 单个分片负载过高，成为热点 |
| **扩展性差** | 数据增长后无法水平扩展 |
| **恢复慢** | 单个分片过大，恢复时间长 |

### 4.2 分片过少的判断

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     分片过少的表现                                        │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   • 单分片大小 > 100GB                                                   │
│   • 单分片文档数 > 2 亿                                                  │
│   • 查询时部分节点负载过高                                               │
│   • 节点重启后恢复时间过长                                               │
│   • 数据节点 CPU/内存利用率不均衡                                        │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 五、不同场景的分片策略

### 5.1 场景化建议

| 场景 | 数据量 | 分片大小 | 分片数量 | 副本数 |
|------|--------|----------|----------|--------|
| **日志/监控** | 大 | 30-50GB | 按天/周滚动 | 1 |
| **电商商品** | 中 | 10-30GB | 3-5 个/索引 | 1-2 |
| **用户数据** | 中 | 10-30GB | 按用户量估算 | 2 |
| **金融交易** | 中 | 10-20GB | 按业务量估算 | 2-3 |
| **企业搜索** | 小 | 1-10GB | 1-3 个/索引 | 1 |
| **文档检索** | 大 | 20-50GB | 按文档量估算 | 1 |

### 5.2 时间序列数据策略

对于日志、监控等时间序列数据，推荐使用 **Data Streams + ILM** 策略：

```json
// 创建索引生命周期策略
PUT /_ilm/policy/logs-policy
{
  "policy": {
    "phases": {
      "hot": {
        "min_age": "0ms",
        "actions": {
          "rollover": {
            "max_size": "50GB",
            "max_age": "1d"
          },
          "set_priority": {
            "priority": 100
          }
        }
      },
      "warm": {
        "min_age": "7d",
        "actions": {
          "shrink": {
            "number_of_shards": 1
          },
          "forcemerge": {
            "max_num_segments": 1
          },
          "set_priority": {
            "priority": 50
          }
        }
      },
      "cold": {
        "min_age": "30d",
        "actions": {
          "freeze": {},
          "set_priority": {
            "priority": 0
          }
        }
      },
      "delete": {
        "min_age": "90d",
        "actions": {
          "delete": {}
        }
      }
    }
  }
}

// 创建数据流模板
PUT /_index_template/logs-template
{
  "index_patterns": ["logs-*"],
  "template": {
    "settings": {
      "number_of_shards": 3,
      "number_of_replicas": 1,
      "lifecycle.name": "logs-policy"
    }
  },
  "data_stream": {}
}
```

### 5.3 大索引分片策略

对于数据量特别大的索引：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     大索引分片策略                                        │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   数据量：1TB                                                           │
│                                                                         │
│   方案一：单索引多分片                                                   │
│   ├── 分片大小：30GB                                                    │

│   └── 问题：分片数较多，管理开销大                                       │
│                                                                         │

│   ├── 按月创建索引                                                       │

│   ├── 每个索引 3 个分片，每个约 28GB                                     │
│   └── 优点：分片数量合理，便于管理                                       │

│   方案三：按业务拆分                                                      │
│   ├── 按业务类型创建不同索引                                             │
│   ├── 每个索引独立规划分片                                               │
│   └── 优点：业务隔离，互不影响                                           │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

## 六、分片设置示例

### 6.1 创建索引时设置分片

```json
// 小型索引（< 100GB）
PUT /small_index
{
  "settings": {
    "number_of_shards": 3,
    "number_of_replicas": 1
  }
}

// 中型索引（100GB - 500GB）
PUT /medium_index
{
  "settings": {
    "number_of_shards": 10,
    "number_of_replicas": 1
  }
}

// 大型索引（> 500GB，建议拆分）
PUT /large_index_2024_01
{
  "settings": {
    "number_of_shards": 5,
    "number_of_replicas": 1
  }
}
```

### 6.2 动态调整副本数

```json
// 副本数可以动态调整
PUT /my_index/_settings
{
  "number_of_replicas": 2
}

// 批量写入时临时关闭副本
PUT /my_index/_settings
{
  "number_of_replicas": 0
}

// 写入完成后恢复副本
PUT /my_index/_settings
{
  "number_of_replicas": 1
}
```

### 6.3 分片分配控制

```json
// 限制单节点分片数
PUT /my_index/_settings
{
  "index.routing.allocation.total_shards_per_node": 2
}

// 指定分片分配到特定节点
PUT /my_index/_settings
{
  "index.routing.allocation.include._name": "node-1,node-2"
}

// 排除特定节点
PUT /my_index/_settings
{
  "index.routing.allocation.exclude._name": "node-3"
}
```

### 7.3 分片过多时的解决方案

```json
// 1. 关闭不需要的索引
POST /old_index/_close

// 2. 删除过期索引
DELETE /old_index_*

// 3. 合并小索引
POST /_reindex
{
  "source": {
    "index": "small_index_1,small_index_2"
  },
  "dest": {
    "index": "merged_index"
  }
}

// 4. 使用 shrink API 缩减分片
POST /my_index/_shrink/shrunk_index
{
  "settings": {
    "number_of_shards": 1,
    "number_of_replicas": 1
  }
```

## 八、总结

### 8.1 核心原则速查

| 原则 | 建议值 | 原因 |
|------|--------|------|
| 单分片大小 | 10-50GB | 平衡性能与恢复速度 |
| 单分片文档数 | < 2 亿 | 避免单个分片过大 |
| 每 GB 堆内存分片数 | < 20 个 | 避免内存压力 |
| 单节点分片数 | < 1000 个 | 避免资源耗尽 |
| 副本数 | 1-2 个 | 高可用与性能平衡 |

### 8.2 分片设置决策流程

```
┌─────────────────────────────────────────────────────────────────────────┐
│                     分片设置决策流程                                      │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   1. 估算数据总量                                                        │
│      │                                                                  │
│      ▼                                                                  │
│   2. 确定单分片目标大小（10-50GB）                                       │
│      │                                                                  │
│      ▼                                                                  │
│   3. 计算分片数量 = 数据总量 / 单分片大小                                 │
│      │                                                                  │
│      ▼                                                                  │
│   4. 评估集群资源（节点数、内存、磁盘）                                   │
│      │                                                                  │
│      ├── 资源充足 → 按计算结果设置                                       │
│      │                                                                  │
│      └── 资源有限 → 减少分片数或增加节点                                  │
│             │                                                           │
│             ▼                                                           │
│   5. 设置副本数（生产环境至少 1 个）                                      │
│      │                                                                  │
│      ▼                                                                  │
│   6. 监控并动态调整                                                       │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```
