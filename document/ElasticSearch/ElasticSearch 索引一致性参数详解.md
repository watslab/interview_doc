# ElasticSearch 索引一致性参数详解

## 概述

ElasticSearch 提供了写入一致性控制机制，确保数据在返回成功前已写入到足够数量的分片，从而在性能与数据安全之间取得平衡。

---

## 一、一致性参数概述

### 版本差异

| ES 版本 | 参数名称 | 状态 |
|---------|----------|------|
| ES 5.0 之前 | `consistency` | 已废弃 |
| ES 5.0+ | `wait_for_active_shards` | 推荐使用 |

---

## 二、consistency 参数（已废弃）

### 参数值说明

| 值 | 说明 | 计算方式 |
|----|------|----------|
| `one` | 只要主分片活跃即可写入 | 最低一致性 |
| `quorum`（默认） | 需要大多数分片活跃 | `(分片数/2) + 1` |
| `all` | 需要所有分片（主+副本）都活跃 | 最高一致性 |

### 使用示例

```json
// 使用 consistency 参数（已废弃）
PUT /my_index/_doc/1?consistency=one
{
  "title": "文档标题"
}

PUT /my_index/_doc/1?consistency=quorum
{
  "title": "文档标题"
}

PUT /my_index/_doc/1?consistency=all
{
  "title": "文档标题"
}
```

---

## 三、wait_for_active_shards 参数（推荐）

### 参数值说明

| 值 | 说明 |
|----|------|
| `1`（默认） | 只需主分片写入成功即可返回 |
| `2` | 需要主分片 + 1个副本分片写入成功 |
| `3` | 需要主分片 + 2个副本分片写入成功 |
| `all` | 需要主分片 + 所有副本分片写入成功 |

### 使用示例

#### 单次请求设置

```json
// 只需主分片成功（默认）
PUT /my_index/_doc/1?wait_for_active_shards=1
{
  "title": "文档标题"
}

// 需要主分片 + 1个副本成功
PUT /my_index/_doc/1?wait_for_active_shards=2
{
  "title": "文档标题"
}

// 需要所有分片成功
PUT /my_index/_doc/1?wait_for_active_shards=all
{
  "title": "文档标题"
}
```

#### 索引级别设置

```json
// 创建索引时设置默认一致性
PUT /my_index
{
  "settings": {
    "number_of_shards": 3,
    "number_of_replicas": 1,
    "index.write.wait_for_active_shards": "all"
  }
}

// 动态修改索引的一致性设置
PUT /my_index/_settings
{
  "index.write.wait_for_active_shards": 2
}
```

#### 集群级别设置

```json
// 设置集群默认一致性
PUT /_cluster/settings
{
  "persistent": {
    "index.write.wait_for_active_shards": 1
  }
}
```

## 五、最佳实践

### 5.1 场景化建议

| 业务场景 | 推荐设置 | 理由 |
|----------|----------|------|
| **日志收集** | `1` | 可容忍少量数据丢失，追求高性能 |
| **监控指标** | `1` | 数据量大，可容忍丢失 |
| **电商商品** | `2` | 平衡性能与数据安全 |
| **社交动态** | `2` | 平衡性能与数据安全 |
| **用户账户** | `all` | 数据不能丢失 |
| **金融交易** | `all` | 数据绝对安全 |
| **订单支付** | `all` | 数据绝对安全 |

### 5.2 配置示例

#### 通用业务索引（推荐配置）

```json
PUT /my_index
{
  "settings": {
    "number_of_shards": 3,
    "number_of_replicas": 1,
    "index.write.wait_for_active_shards": 2
  }
}
```

#### 关键业务索引（高一致性）

```json
PUT /payment_index
{
  "settings": {
    "number_of_shards": 3,
    "number_of_replicas": 2,
    "index.write.wait_for_active_shards": "all"
  }
}
```

#### 日志索引（高性能）

```json
PUT /log_index
{
  "settings": {
    "number_of_shards": 5,
    "number_of_replicas": 1,
    "index.write.wait_for_active_shards": 1
  }
}
```

### 5.3 注意事项

| 注意点 | 说明 |
|--------|------|
| **副本数影响** | `wait_for_active_shards` 的值不能超过 `number_of_replicas + 1` |
| **性能影响** | 值越大，写入延迟越高 |
| **可用性影响** | 值越大，需要更多节点在线才能写入 |
| **与副本数配合** | 设置 `all` 时，建议副本数 ≥ 2 |

### 5.4 异常处理

#### 常见错误

```json
// 当活跃分片不足时，写入会失败
{
  "error": {
    "type": "unavailable_shards_exception",
    "reason": "[my_index][0] Not enough active copies to meet write consistency..."
  },
  "status": 503
}
```

#### 排查步骤

```json
// 1. 检查集群健康状态
GET /_cluster/health

// 2. 检查分片状态
GET /_cat/shards?v&h=index,shard,prirep,state

// 3. 检查节点状态
GET /_cat/nodes?v
```

#### 临时解决方案

```json
// 临时降低一致性要求
PUT /my_index/_doc/1?wait_for_active_shards=1
{
  "title": "临时写入"
}
```

## 七、与其他参数的关系

### 与 number_of_replicas 的关系

```
┌─────────────────────────────────────────────────────────────────────────┐
│           wait_for_active_shards 与 number_of_replicas 的关系           │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   number_of_replicas = 1（1主 + 1副本 = 2个分片）                        │
│   ├── wait_for_active_shards = 1 → 只需主分片                          │
│   ├── wait_for_active_shards = 2 → 主分片 + 1个副本                    │
│   └── wait_for_active_shards = all → 主分片 + 所有副本 = 2             │
│                                                                         │
│   number_of_replicas = 2（1主 + 2副本 = 3个分片）                        │
│   ├── wait_for_active_shards = 1 → 只需主分片                          │
│   ├── wait_for_active_shards = 2 → 主分片 + 1个副本                    │
│   ├── wait_for_active_shards = 3 → 主分片 + 2个副本                    │
│   └── wait_for_active_shards = all → 主分片 + 所有副本 = 3             │
│                                                                         │
│   【重要】wait_for_active_shards 不能超过总分片数                        │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 与 refresh_interval 的关系

| 参数 | 作用 | 影响 |
|------|------|------|
| `wait_for_active_shards` | 控制写入成功前需要同步的分片数 | 影响写入延迟 |
| `refresh_interval` | 控制数据何时可被搜索 | 影响搜索可见性 |

两者独立，`wait_for_active_shards` 只影响写入成功返回的时机，不影响数据可搜索的时机。

## 八、总结

### 配置建议速查表

| 配置项 | 推荐值 | 适用场景 |
|--------|--------|----------|
| `wait_for_active_shards` | `1` | 日志、监控等可容忍丢失的场景 |
| `wait_for_active_shards` | `2`（推荐） | 大多数业务场景 |
| `wait_for_active_shards` | `all` | 金融、支付等关键业务 |

### 核心原则

1. **根据业务需求选择**：在性能与数据安全之间找到平衡点
2. **与副本数配合**：副本数越多，可设置的一致性级别越高
3. **监控集群状态**：确保有足够的活跃分片满足一致性要求
4. **灵活调整**：可根据实际情况动态调整一致性级别
