## 一、集群架构与节点管理

### 1.1 集群基本概念

| 概念 | 说明 |
|------|------|
| **Cluster（集群）** | 一组具有相同 cluster.name 的节点集合 |
| **Node（节点）** | 一个运行中的 ElasticSearch 实例 |
| **Index（索引）** | 文档的逻辑集合，相当于关系数据库的表 |
| **Type（类型）** | ES 7.x 后已废弃，之前相当于表的逻辑分组 |
| **Document（文档）** | 最小数据单元，JSON 格式，相当于关系数据库的行 |
| **Shard（分片）** | 索引的物理存储单元，分为主分片和副本分片 |
| **Replica（副本）** | 主分片的复制，提供高可用和查询负载均衡 |

### 1.2 脑裂问题（Split Brain）

#### 什么是脑裂？

脑裂是指集群中出现多个 Master 节点，导致集群状态不一致的问题。这是分布式系统中最危险的问题之一。

```
┌─────────────────────────────────────────────────────────────────────────┐
│                           脑裂问题示意                                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   正常情况：                                                             │
│   ┌─────────────────────────────────────────────────────┐              │
│   │              Cluster (1 Master)                      │              │
│   │   ┌─────────┐  ┌─────────┐  ┌─────────┐             │              │
│   │   │ Master  │  │  Data   │  │  Data   │             │              │
│   │   │  Node   │  │  Node   │  │  Node   │             │              │
│   │   └─────────┘  └─────────┘  └─────────┘             │              │
│   └─────────────────────────────────────────────────────┘              │
│                                                                         │
│   脑裂情况：                                                             │
│   ┌────────────────────┐        ┌────────────────────┐                 │
│   │  Cluster A         │        │  Cluster B         │                 │
│   │  (Master-1)        │   ✕    │  (Master-2)        │  网络分区       │
│   │  ┌─────────┐       │        │       ┌─────────┐  │                 │
│   │  │ Master  │       │        │       │ Master  │  │                 │
│   │  │  Data   │       │        │       │  Data   │  │                 │
│   │  └─────────┘       │        │       └─────────┘  │                 │
│   └────────────────────┘        └────────────────────┘                 │
│                                                                         │
│   后果：数据不一致、写入冲突、分片分配混乱、数据丢失                       │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

#### 脑裂产生的原因

| 原因 | 说明 |
|------|------|
| **网络分区** | 网络故障导致节点之间无法通信 |
| **节点故障** | Master 节点负载过高或崩溃 |
| **配置错误** | minimum_master_nodes 配置不当 |
| **资源耗尽** | 内存、CPU、磁盘等资源耗尽导致节点无响应 |

#### 脑裂的危害

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          脑裂的危害                                      │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   1. 数据不一致                                                         │
│      • 两个集群独立接受写入，数据产生分歧                                │
│      • 网络恢复后无法自动合并                                            │
│                                                                         │
│   2. 数据丢失                                                           │
│      • 分片分配混乱，部分数据无法访问                                    │
│      • 主分片冲突，需要手动选择保留哪个版本                              │
│                                                                         │
│   3. 服务中断                                                           │
│      • 客户端连接到错误的集群                                            │
│      • 查询结果不完整或错误                                              │
│                                                                         │
│   4. 恢复困难                                                           │
│      • 需要手动干预恢复                                                  │
│      • 可能需要重建索引                                                  │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

#### ES 6.x 及之前版本的解决方案

在 ES 6.x 及之前版本，通过配置 `discovery.zen.minimum_master_nodes` 来防止脑裂。

```yaml
# elasticsearch.yml（ES 6.x）

# 最小主节点数
discovery.zen.minimum_master_nodes: 2

# 计算公式：(master候选节点数 / 2) + 1
# 例如：3个master节点 → (3/2) + 1 = 2
# 例如：5个master节点 → (5/2) + 1 = 3
```

**原理**：只有获得超过半数 Master 节点的投票，才能成为 Master。

---

#### ES 7.x+ 版本的解决方案（重点）

ES 7.x 对集群发现机制进行了重大改进，引入了新的配置方式，**自动计算 minimum_master_nodes**，大大降低了脑裂风险。

##### 1. 新增配置项

| 配置项 | 说明 | 示例 |
|--------|------|------|
| `cluster.initial_master_nodes` | 首次启动时的初始主节点候选列表 | `["node-1", "node-2", "node-3"]` |
| `discovery.seed_hosts` | 种子节点列表，用于发现其他节点 | `["192.168.1.101", "192.168.1.102"]` |
| `discovery.type` | 发现类型，默认 `zen` | `zen` / `single-node` |

##### 2. 完整配置示例

```yaml
# elasticsearch.yml（ES 7.x+）

# 集群名称（所有节点必须一致）
cluster.name: my-es-cluster

# 节点名称（每个节点唯一）
node.name: node-1

# 节点角色
node.roles: [master, data]

# 网络配置
network.host: 0.0.0.0
http.port: 9200
transport.port: 9300

# 【关键】种子节点列表（用于发现其他节点）
discovery.seed_hosts: ["192.168.1.101", "192.168.1.102", "192.168.1.103"]

# 【关键】初始主节点候选列表（首次启动时使用）
# 只在集群首次启动时需要，之后会自动持久化
cluster.initial_master_nodes: ["node-1", "node-2", "node-3"]
```

##### 3. ES 7.x+ 防脑裂机制

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    ES 7.x+ 防脑裂机制                                    │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   改进点 1：自动计算 minimum_master_nodes                                │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │ • ES 7.x+ 自动根据 cluster.initial_master_nodes 的数量计算       │   │
│   │ • 无需手动配置 discovery.zen.minimum_master_nodes               │   │
│   │ • 避免人为配置错误导致的脑裂                                      │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│   改进点 2：Bootstrap 检查                                              │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │ • 首次启动时必须指定 cluster.initial_master_nodes               │   │
│   │ • 防止意外启动多个独立集群                                        │   │

│                                                                         │
```

##### 5. ES 7.x+ 与 ES 6.x 配置对比

| 对比项 | ES 6.x | ES 7.x+ |
|--------|--------|---------|
| 最小主节点数 | 手动配置 `discovery.zen.minimum_master_nodes` | **自动计算** |
| 初始主节点 | 无 | `cluster.initial_master_nodes` |
| 种子节点 | `discovery.zen.ping.unicast.hosts` | `discovery.seed_hosts` |
| 单节点模式 | `discovery.zen.minimum_master_nodes: 1` | `discovery.type: single-node` |
| 首次启动检查 | 无强制要求 | **必须配置 initial_master_nodes** |

#### 集群状态（Cluster State）详解

##### 什么是集群状态？

集群状态（Cluster State）是 ElasticSearch 集群的核心数据结构，每个节点都会在内存中维护一份当前的集群状态副本。Master 节点负责维护和更新集群状态，并同步给其他节点。

#### discovery.seed_hosts 配置详解

##### 是否必须配置所有集群节点？

**答案：不需要。** `discovery.seed_hosts` 只需要配置**部分节点**即可，不必配置所有节点。

##### 配置原则

| 原则 | 说明 |
|------|------|
| **最少配置** | 至少配置一个可达的 Master 候选节点 |
| **推荐配置** | 配置所有 Master 候选节点 |
| **最佳实践** | 配置 3-5 个稳定的 Master 节点地址 |

##### 工作原理

```
┌─────────────────────────────────────────────────────────────────────────┐
│                   discovery.seed_hosts 工作原理                          │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   节点启动时：                                                           │
│   1. 读取 discovery.seed_hosts 配置                                    │
│      │                                                                  │
│      ▼                                                                  │
│   2. 向 seed_hosts 中的节点发送发现请求                                  │
│      │                                                                  │
│      ▼                                                                  │
│   3. 获取集群状态，发现其他节点                                          │
│      │                                                                  │
│      ▼                                                                  │
│   4. 加入集群                                                           │
│                                                                         │
│   【说明】                                                               │
│   • seed_hosts 是"种子节点"，用于引导发现                                │
│   • 通过种子节点可以获取完整的节点列表                                    │
│   • 不需要配置所有节点，但建议配置多个以保证可用性                         │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

##### 配置示例

```yaml
# 方式一：配置所有 Master 节点（推荐）
discovery.seed_hosts: ["192.168.1.101", "192.168.1.102", "192.168.1.103"]

# 方式二：配置部分 Master 节点（最少一个）
discovery.seed_hosts: ["192.168.1.101", "192.168.1.102"]

# 方式三：配置单个 Master 节点（不推荐，单点故障风险）
discovery.seed_hosts: ["192.168.1.101"]

# 方式四：指定端口（默认9300）
discovery.seed_hosts: ["192.168.1.101:9300", "192.168.1.102:9300"]
```

##### 注意事项

| 注意点 | 说明 |
|--------|------|
| **高可用** | 建议配置多个 Master 节点，避免单点故障 |
| **端口** | 默认使用 transport.port（9300），非 HTTP 端口（9200） |
| **新节点加入** | 新节点只需配置一个可达的 seed_hosts 即可加入集群 |
| **动态发现** | 通过 seed_hosts 获取的节点信息会缓存，后续可发现新节点 |

#### cluster.initial_master_nodes 配置详解

##### 是否必须配置所有 Master 节点？

**答案：不需要。** 但有重要限制条件。

##### 配置规则

| 场景 | 是否必须配置所有 Master 节点 | 说明 |
|------|------------------------------|------|
| **首次启动集群** | 否，但建议配置 | 只需配置参与首次选主的节点 |
| **集群已形成** | 不需要配置 | 集群状态已持久化，此配置无效 |
| **新节点加入已有集群** | 不需要配置 | 新节点会自动加入 |

##### 如果 Master 节点未配置在 initial_master_nodes 中会怎样？

```
┌─────────────────────────────────────────────────────────────────────────┐
│         initial_master_nodes 未包含所有 Master 节点的影响                │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   场景：5个 Master 节点，但只配置了 3 个                                  │
│                                                                         │
│   配置：                                                                 │
│   cluster.initial_master_nodes: ["node-1", "node-2", "node-3"]         │
│                                                                         │
│   实际节点：node-1, node-2, node-3, node-4, node-5                      │
│   （node-4 和 node-5 的 roles 包含 master，但未在列表中）                │
│                                                                         │
│   结果：                                                                 │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │ 1. node-1, node-2, node-3 参与首次选主                          │   │
│   │ 2. node-4, node-5 不参与首次选主                                │   │
│   │ 3. 集群形成后，node-4, node-5 自动加入集群                      │   │
│   │ 4. node-4, node-5 可以参与后续的选主投票                        │   │
│   │ 5. minimum_master_nodes = (3/2) + 1 = 2（基于 initial_master）  │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
│   【重要】minimum_master_nodes 基于 initial_master_nodes 的数量计算     │
│   如果后续添加更多 Master 节点，需要手动更新 minimum_master_nodes        │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

##### 最佳实践

```yaml
# 推荐做法：配置所有 Master 候选节点
cluster.initial_master_nodes: ["master-1", "master-2", "master-3"]

# 不推荐做法：只配置部分 Master 节点
# cluster.initial_master_nodes: ["master-1", "master-2"]
# 问题：minimum_master_nodes 计算不准确
```

##### 集群形成后的处理

```bash
# 集群首次启动成功后，建议移除 initial_master_nodes 配置
# 因为集群状态已持久化，此配置不再需要

# 查看集群是否已形成
GET /_cluster/state

# 如果集群已形成，可以从配置文件中移除 cluster.initial_master_nodes
# 重启节点不影响集群运行
```

##### 手动更新 minimum_master_nodes

如果后续添加了新的 Master 节点，需要手动更新：

```json
# 添加新 Master 节点后，更新 minimum_master_nodes
PUT /_cluster/settings
{
  "persistent": {
    "discovery.zen.minimum_master_nodes": 3
  }
}
```

##### 配置对比总结

| 配置项 | 是否必须配置所有节点 | 建议 |
|--------|----------------------|------|
| `discovery.seed_hosts` | 否，配置部分即可 | 配置所有 Master 节点 |
| `cluster.initial_master_nodes` | 否，但影响 minimum_master_nodes 计算 | 配置所有 Master 候选节点 |

### 1.4 集群发现机制（Discovery）

#### 发现机制原理

集群发现是节点之间相互发现并组成集群的过程。

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         集群发现机制                                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   1. 节点启动                                                           │
│      │                                                                  │
│      ▼                                                                  │
│   2. 查找 discovery.seed_hosts 中的节点                                 │
│      │                                                                  │
│      ▼                                                                  │
│   3. 获取集群状态（Cluster State）                                      │
│      │                                                                  │
│      ├── 如果发现已有 Master → 加入集群                                 │
│      │                                                                  │
│      └── 如果没有 Master → 参与选主                                     │
│               │                                                         │
│               ▼                                                         │
│   4. 选主成功后，同步集群状态                                            │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

#### 发现相关配置

```yaml
# elasticsearch.yml
```

### 1.6 路由机制（Routing）

#### 什么是路由？

路由决定文档存储在哪个分片上。

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         文档路由机制                                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   文档写入请求                                                          │
│        │                                                                │
│        ▼                                                                │
│   ┌─────────────────────────────────────────────────────────────────┐   │
│   │  计算路由：shard_num = hash(_routing) % num_primary_shards      │   │
│   │                                                                  │   │
│   │  默认 _routing = _id                                            │   │
│   └─────────────────────────────────────────────────────────────────┘   │
│        │                                                                │
│        ▼                                                                │
│   确定目标分片                                                          │
│        │                                                                │
│        ▼                                                                │
│   写入主分片 → 同步到副本分片                                            │
│                                                                         │
│   示例：                                                                 │
│   • 文档 ID: "doc-123"                                                  │
│   • 主分片数: 3                                                         │
│   • hash("doc-123") % 3 = 1 → 存储在 P1 分片                           │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

#### 自定义路由

```json
# 写入时指定路由
PUT /my_index/_doc/1?routing=user_123
{
  "title": "文档标题",
  "user_id": "user_123"
}

# 查询时指定路由（只查询特定分片，提高性能）
GET /my_index/_search?routing=user_123
{
  "query": {
    "match": { "title": "文档" }
  }
}

# 索引映射中配置路由必需
PUT /my_index
{
  "mappings": {
    "_routing": {
      "required": true
    }
  }
}
```

### 1.7 分片重平衡（Rebalance）

#### 什么是 Rebalance？

Rebalance 是自动调整分片分布的过程，确保各节点负载均衡。

```
┌─────────────────────────────────────────────────────────────────────────┐
│                       分片重平衡示意                                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   重平衡前（Node-3 负载高）：                                            │
│   ┌─────────────┐   ┌─────────────┐   ┌─────────────┐                  │
│   │   Node-1    │   │   Node-2    │   │   Node-3    │                  │
│   │   P0, R1    │   │   P1, R2    │   │   P2, R0    │                  │
│   │   (2分片)   │   │   (2分片)   │   │   P3, R4    │ ← 负载高         │
│   │             │   │             │   │   P5, R6    │                  │
│   │             │   │             │   │   (6分片)   │                  │
│   └─────────────┘   └─────────────┘   └─────────────┘                  │
│                                                                         │
│   重平衡后（自动迁移分片）：                                              │
│   ┌─────────────┐   ┌─────────────┐   ┌─────────────┐                  │
│   │   Node-1    │   │   Node-2    │   │   Node-3    │                  │
│   │   P0, R1    │   │   P1, R2    │   │   P2, R0    │                  │
│   │   P3        │   │   R4        │   │   P5        │                  │
│   │   (3分片)   │   │   (3分片)   │   │   R6        │                  │
│   │             │   │             │   │   (4分片)   │                  │
│   └─────────────┘   └─────────────┘   └─────────────┘                  │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

#### Rebalance 配置

```json

{

    "cluster.routing.rebalance.enable": "all",
```

## 二、分片管理

### 2.1 分片规划原则

| 原则 | 说明 |
|------|------|
| 分片大小 | 单个分片建议 10-50GB |
| 分片数量 | 每个节点不超过 20 个分片/GB 堆内存 |
| 主分片数 | 创建后不可修改 |
| 副本数 | 可动态调整 |

## 八、最佳实践总结

### 8.1 集群规划

| 实践 | 说明 |
|------|------|
| 主节点数 | 奇数个，至少 3 个（避免脑裂） |
| 数据节点 | 根据数据量和查询负载规划 |
| 专用节点 | 生产环境建议角色分离 |
| 机架感知 | 避免单点故障 |

### 8.2 分片规划

| 实践 | 说明 |
|------|------|
| 分片大小 | 10-50GB |
| 分片数量 | 每节点 ≤ 20 个分片/GB 堆内存 |
| 副本数 | 生产环境 ≥ 1 |
| 预分配 | 根据数据增长提前规划 |

### 8.3 监控告警

| 实践 | 说明 |
|------|------|
| 全面监控 | 集群、节点、索引、查询 |
| 合理告警 | 设置合适的阈值 |
| 快速响应 | 建立响应机制 |
| 日志分析 | 定期分析日志 |

### 8.4 备份策略

| 实践 | 说明 |
|------|------|
| 定期备份 | 每日增量 + 每周全量 |
| 异地备份 | 灾难恢复保障 |
| 定期演练 | 验证备份可恢复性 |
| 保留策略 | 根据合规要求设置 |
