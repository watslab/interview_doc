# ElasticSearch 数值类型索引结构详解

## 概述

ElasticSearch 中不同数据类型使用不同的索引结构，数值类型使用 **BKD 树**而非传统的倒排索引。

---

## 一、不同数据类型的底层索引结构

| 数据类型 | 底层索引结构 | 说明 |
|----------|--------------|------|
| `text`、`keyword` | **倒排索引** + Doc Values | 分词或不分词的倒排列表 |
| `integer`、`long`、`float`、`double` | **BKD 树** | 数值类型专用 |
| `date` | **BKD 树** | 时间戳作为数值存储 |
| `geo_point`、`geo_shape` | **BKD 树** | 地理坐标作为多维数值 |
| `boolean` | **倒排索引** | 只有 T/F 两个 term |
| `range` | **BKD 树** | 存储为低维 BKD 点 |
| `binary` | Doc Values | 仅存储，不索引 |

---

## 二、为什么数值类型不用倒排索引？

### 倒排索引的问题

倒排索引适合**精确匹配**，但对数值类型的**范围查询**效率低下：

```json
// 范围查询：查找年龄 20-30 的文档
GET /users/_search
{
  "query": {
    "range": {
      "age": { "gte": 20, "lte": 30 }
    }
  }
}
```

如果用倒排索引，需要：
1. 遍历所有 term（20, 21, 22, ... 30）
2. 合并每个 term 的倒排列表
3. 性能极差

### BKD 树的优势

BKD 树（Block K-D Tree）是专门为**数值范围查询**优化的数据结构：

| 特性 | 说明 |
|------|------|
| 范围查询 | O(log N) 时间复杂度 |
| 多维支持 | 支持地理坐标等多维数据 |
| 空间效率 | 块状存储，节省空间 |
| 排序友好 | 天然有序，适合排序 |

---

## 三、BKD 树工作原理

### 核心结构

BKD 树将数值按范围切分为多个 **Block（块）**，每个 Block 包含：

- 有序的数值（如时间戳递增排列）
- 对应数值的文档 ID（docid）列表

树的上层节点记录每个 Block 的数值范围，查询时先通过上层节点定位目标 Block，再扫描 Block 内数据。

### 示例：日期范围查询

假设有 100 万文档，`publish_time` 为 date 类型：

```
Block A：1748400000 ~ 1748450000 → docid 列表：[5, 2, 9, 13, 7, ...]
Block B：1748450001 ~ 1748500000 → docid 列表：[31, 18, 45, 22, ...]
Block C：1748500001 ~ 1748550000 → docid 列表：[56, 72, 39, 88, ...]
```

### 查询执行流程

1. **定位目标 Block**：遍历 BKD 树上层节点，快速匹配与查询范围重叠的 Block
2. **提取匹配的 docid**：遍历每个 Block，筛选出符合条件的 docid
3. **构造 Bitset**：为每个 Block 构造位图，通过位运算合并结果

## 四、BKD 树 vs 倒排索引对比

| 对比项 | 倒排索引 | BKD 树 |
|--------|----------|--------|
| 适用类型 | 字符串 | 数值、日期、地理 |
| 精确匹配 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 范围查询 | ⭐⭐ | ⭐⭐⭐⭐⭐ |
| 排序 | ⭐⭐ | ⭐⭐⭐⭐⭐ |
| 聚合 | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 前缀/模糊查询 | ⭐⭐⭐⭐⭐ | ❌ 不支持 |

## 五、index 参数的作用

### 核心概念

`index` 参数控制的是**是否创建索引**，而不是**创建什么类型的索引**。

| 参数值 | 作用 |
|--------|------|
| `index=true`（默认） | 创建索引，字段可被搜索 |
| `index=false` | 不创建索引，字段不可被搜索 |

### 索引类型由字段类型决定

**索引的类型是由字段的数据类型决定的**，而不是 `index` 参数：

```json
PUT /my_index
{
  "mappings": {
    "properties": {
      "title": {
        "type": "text",
        "index": true          // 创建倒排索引
      },
      "age": {
        "type": "integer",
        "index": true          // 创建 BKD 树索引
      },
      "price": {
        "type": "double",
        "index": false         // 不创建任何索引
      }
    }
  }
}
```

| 字段类型 | `index=true` 创建的索引 |
|----------|------------------------|
| `text` / `keyword` | 倒排索引 |
| `integer` / `long` / `float` / `double` | **BKD 树** |
| `date` | **BKD 树** |
| `geo_point` | **BKD 树** |
| `boolean` | 倒排索引 |

## 六、完整的索引配置示例

```json
PUT /products
{
  "mappings": {
    "properties": {
      "name": {
        "type": "text",
        "index": true           // 倒排索引，支持全文搜索
      },
      "price": {
        "type": "double",
        "index": true,          // BKD 树，支持范围查询
        "doc_values": true      // Doc Values，支持排序和聚合
      },
      "stock": {
        "type": "integer",
        "index": false,         // 不创建BKD树，不可搜索
        "doc_values": true      // 但仍可通过Doc Values排序/聚合
      },
      "description": {
        "type": "text",
        "index": false          // 不创建倒排索引，不可搜索
      },
      "created_at": {
        "type": "date",
        "index": true           // BKD 树，支持时间范围查询
      },
      "location": {
        "type": "geo_point",
        "index": true           // BKD 树，支持地理距离查询
      }
    }
  }
}
```

## 八、常见问题

### Q1: 数值类型可以创建倒排索引吗？

**答**：不可以。数值类型固定使用 BKD 树，这是由 Lucene 底层实现的。

### Q2: index=false 的字段还能排序和聚合吗？

**答**：可以。只要 `doc_values=true`（默认开启），即使 `index=false`，字段仍可进行排序和聚合操作。

### Q3: BKD 树的查询性能如何？

**答**：
- 范围查询：O(log N)，性能优秀
- 精确匹配：略逊于倒排索引，但仍可接受
- 大范围查询：可能涉及 Bitset 构造开销

### Q4: 如何优化数值字段的查询性能？

**答**：
1. 使用时间分片索引，减少单索引数据量
2. 调整 BKD 树 Block 大小
3. 利用缓存优化 IO
4. 预过滤缩小 docid 范围
