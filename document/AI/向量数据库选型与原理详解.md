# 向量数据库选型与原理详解

## 一、概述

向量数据库是专门用于存储和查询向量数据（高维数值数组）的数据库系统，它通过高效的相似性搜索算法，解决了传统数据库无法处理的"语义匹配"问题。

### 1.1 为什么需要向量数据库？

| 场景 | 传统数据库 | 向量数据库 |
|------|-----------|-----------|
| **搜索方式** | 关键词精确匹配 | 语义相似度匹配 |
| **查询类型** | 结构化查询（SQL） | 向量相似性查询 |
| **数据类型** | 表格、文档 | 高维向量（Embedding） |
| **典型应用** | 业务系统、事务处理 | AI 应用、RAG、推荐系统 |

## 二、核心原理

### 2.1 向量相似度计算

```mermaid
flowchart TB
    subgraph Distance["相似度计算方法"]
        D1["欧氏距离 L2<br/>几何距离"]
        D2["余弦相似度<br/>方向相似性"]
        D3["点积 IP<br/>投影相似性"]
    end
    
    subgraph Formula["计算公式"]
        F1["L2 = √Σ(ai - bi)²"]
        F2["Cosine = A·B / |A||B|"]
        F3["IP = A·B"]
    end
    
    Distance --> Formula
    
    style Distance fill:#e3f2fd,stroke:#1565c0
    style Formula fill:#fff3e0,stroke:#ef6c00
```

| 方法 | 公式 | 适用场景 | 特点 |
|------|------|---------|------|
| **欧氏距离 (L2)** | √Σ(ai - bi)² | 图像检索、聚类 | 关注绝对距离 |
| **余弦相似度** | A·B / \|A\|\|B\| | 文本语义、推荐 | 关注方向，忽略长度 |
| **点积 (IP)** | A·B | 归一化向量 | 计算简单 |

### 2.2 向量索引算法

向量索引是向量数据库的核心，决定了检索效率与精度。

#### 2.2.1 索引算法分类

```mermaid

        subgraph ANN["近似最近邻 ANN"]
            IVF["IVF 系列<br/>倒排索引"]
            HNSW["HNSW<br/>分层导航小世界图"]

        end
    end
    
    style Exact fill:#ffcdd2,stroke:#c62828
    style ANN fill:#c8e6c9,stroke:#2e7d32
```

#### 2.2.2 FLAT 索引（暴力搜索）

```mermaid
flowchart LR
    subgraph FLAT["FLAT 索引"]
        Q["查询向量"] --> C["与所有向量计算距离"]
        C --> S["排序返回 Top-K"]
    end
    
    style FLAT fill:#fff3e0,stroke:#ef6c00
```

| 特点 | 说明 |
|------|------|
| **精度** | 100% 召回率 |
| **速度** | 最慢，O(N) 复杂度 |
| **内存** | 需存储全部向量 |
| **适用场景** | 小规模数据（< 10 万）、高精度要求 |

#### 2.2.3 IVF 索引（倒排文件索引）

```mermaid
flowchart TB
    subgraph IVF["IVF 索引原理"]
        subgraph Build["构建阶段"]
            B1["K-Means 聚类"] --> B2["划分 N 个簇"]
            B2 --> B3["每个簇建立倒排列表"]
        end
        
        subgraph Search["查询阶段"]
            S1["计算查询向量与聚类中心距离"] --> S2["选择最近的 n_probe 个簇"]
            S2 --> S3["在这些簇内暴力搜索"]
        end
    end
    
    Build --> Search
    
    style IVF fill:#e3f2fd,stroke:#1565c0
    style Build fill:#c8e6c9,stroke:#2e7d32
    style Search fill:#fff3e0,stroke:#ef6c00
```

**IVF 系列索引对比**：

| 索引类型 | 说明 | 内存占用 | 查询速度 | 召回率 |
|---------|------|---------|---------|--------|
| **IVF_FLAT** | 原始向量存储 | 高 | 中 | 高 |
| **IVF_SQ8** | 8 位标量量化 | 低 | 快 | 中 |
| **IVF_PQ** | 乘积量化压缩 | 最低 | 最快 | 较低 |

**关键参数**：

| 参数 | 说明 | 影响 |
|------|------|------|
| `nlist` | 聚类中心数量 | 越大越精细，但构建越慢 |
| `nprobe` | 查询时搜索的簇数 | 越大召回率越高，速度越慢 |

#### 2.2.4 HNSW 索引（分层导航小世界图）

HNSW（Hierarchical Navigable Small World）是目前生产环境首选索引，兼顾检索速度、召回率与动态更新。

```mermaid
flowchart TB
    subgraph HNSW["HNSW 分层结构"]
        L2["Layer 2<br/>顶层稀疏<br/>全局导航"]
        L1["Layer 1<br/>中间层<br/>区域导航"]
        L0["Layer 0<br/>底层稠密<br/>全量向量"]
    end
    
    L2 --> L1 --> L0
    
    style HNSW fill:#e3f2fd,stroke:#1565c0
    style L2 fill:#ffcdd2,stroke:#c62828
    style L1 fill:#fff3e0,stroke:#ef6c00
    style L0 fill:#c8e6c9,stroke:#2e7d32
```

**工作原理**：

```mermaid
sequenceDiagram
    participant Q as 查询向量
    participant L2 as Layer 2
    participant L1 as Layer 1
    participant L0 as Layer 0
    
    Q->>L2: 从入口点开始
    L2->>L2: 贪婪搜索最近节点
    L2->>L1: 下降到下一层
    L1->>L1: 继续贪婪搜索
    L1->>L0: 下降到底层
    L0->>L0: 精细搜索 ef_search 范围
    L0-->>Q: 返回 Top-K 结果
```

**关键参数**：

| 参数 | 说明 | 建议 |
|------|------|------|
| `M` | 每个节点的最大连接数 | 16-64，越大召回率越高 |
| `ef_construction` | 构建时搜索宽度 | 100-400，越大构建越慢 |
| `ef_search` | 查询时搜索宽度 | 10-100，越大召回率越高 |

**HNSW vs IVF 对比**：

| 对比维度 | HNSW | IVF |
|---------|------|-----|
| **查询速度** | 更快 | 较快 |
| **召回率** | 更高 | 可调 |
| **内存占用** | 较高 | 较低 |
| **构建速度** | 较慢 | 较快 |
| **动态更新** | 支持 | 需重建 |
| **适用场景** | 实时检索、高召回 | 大规模、内存受限 |

| 原始向量 | 压缩后 | 压缩比 |

### 2.3 混合检索

```mermaid
flowchart LR
    subgraph Hybrid["混合检索"]
        V["向量检索<br/>语义相似"]
        K["关键词检索<br/>精确匹配"]
        R["结果融合<br/>重排序"]
    end
    
    V --> R
    K --> R
    
    style Hybrid fill:#e3f2fd,stroke:#1565c0
```

**融合策略**：

| 策略 | 说明 |
|------|------|
| **加权融合** | 向量分数 × α + 关键词分数 × β |
| **倒数排名融合 (RRF)** | rank = Σ 1/(k + rank_i) |
| **重排序** | 先向量粗筛，再关键词精排 |

## 五、RAG 架构实践

### 5.1 RAG 工作流程

```mermaid
sequenceDiagram
    participant User as 用户
    participant App as 应用层
    participant Embed as 嵌入模型
    participant VDB as 向量数据库
    participant LLM as 大模型
    
    Note over User,LLM: 知识库构建阶段
    App->>Embed: 文档分块
    Embed-->>App: 文本向量
    App->>VDB: 存储向量 + 元数据
    
    Note over User,LLM: 查询阶段
    User->>App: 提问
    App->>Embed: 问题向量化
    Embed-->>App: 问题向量
    App->>VDB: 相似度检索
    VDB-->>App: 相关文档片段
    App->>LLM: 问题 + 上下文
    LLM-->>App: 生成回答
    App-->>User: 返回答案
```

### 5.3 最佳实践

| 实践 | 说明 |
|------|------|
| **文档分块** | 推荐 500-1000 tokens，重叠 10-20% |
| **元数据设计** | 包含来源、时间、权限等过滤字段 |
| **索引选择** | 小规模用 HNSW，大规模用 IVF+PQ |
| **混合检索** | 向量 + 关键词，提升召回率 |
| **重排序** | 粗筛 Top-100，精排 Top-10 |

## 七、面试高频问题

| 问题 | 答案要点 |
|------|---------|
| **向量数据库 vs 传统数据库** | 语义匹配 vs 关键词匹配，ANN 索引 vs B+ 树 |
| **HNSW 原理** | 分层图结构，顶层快速导航，底层精细搜索 |
| **IVF 原理** | 聚类划分，倒排索引，nprobe 控制精度 |
| **HNSW vs IVF** | HNSW 更快更高召回，IVF 更省内存 |
| **RAG 中向量数据库作用** | 语义检索、知识存储、幻觉过滤 |
| **如何选择向量数据库** | 数据规模、运维能力、成本预算、功能需求 |
| **混合检索如何实现** | 向量 + 关键词，加权融合或 RRF |
